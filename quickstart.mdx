title: "Quickstart"
description: "Generate your first API client or server stub in under 5 minutes using OpenAPI Generator."
---

This guide walks you through installing OpenAPI Generator, obtaining an OpenAPI spec, and generating both a client SDK and a server stub from the command line.

<Info>
  **Prerequisites**: Node.js 16 or later (for the npm installation method used in this guide). Other installation methods are covered in the [Installation](/installation) page.
</Info>

<Steps>
  <Step title="Install OpenAPI Generator">
    Install the CLI globally via npm:

    ```bash
    npm install @openapitools/openapi-generator-cli -g
    ```

    Verify the installation:

    ```bash
    openapi-generator-cli version
    ```

    You should see the current version printed to your terminal. If you prefer a different installation method (Homebrew, Docker, JAR, etc.), see the [Installation](/installation) page first, then return here.
  </Step>

  <Step title="Get an OpenAPI spec">
    You need an OpenAPI 2.0 or 3.0 spec to generate code from. You can use your own spec or use the Petstore example, which is the canonical demo spec used throughout OpenAPI Generator's documentation.

    Download the Petstore spec locally:

    ```bash
    curl https://raw.githubusercontent.com/openapitools/openapi-generator/master/modules/openapi-generator/src/test/resources/3_0/petstore.yaml \
      -o petstore.yaml
    ```

    The file starts like this:

    ```yaml
    openapi: "3.0.0"
    info:
      version: 1.0.0
      title: Petstore
      license:
        name: MIT
    servers:
      - url: http://petstore.swagger.io/v1
    paths:
      /pets:
        get:
          summary: List all pets
          operationId: listPets
          ...
    ```

    <Tip>
      You can also point the generator directly at a URL without downloading the file first. Pass the URL as the `-i` argument to any `generate` command.
    </Tip>
  </Step>

  <Step title="List available generators">
    Before generating, explore what generators are available:

    ```bash
    openapi-generator-cli list
    ```

    The output groups generators by category:

    ```text
    The following generators are available:

    CLIENT generators:
        ada, apex, bash, c, clojure, cpp-qt-client, cpp-restsdk, cpp-tiny,
        cpp-tizen, cpp-ue4, crystal, csharp, dart, dart-dio, eiffel, elixir,
        elm, erlang-client, erlang-proper, go, groovy, haskell-http-client,
        java, java-helidon-client, java-micronaut-client, javascript,
        javascript-apollo-deprecated, javascript-closure-angular,
        javascript-flowtyped, jaxrs-cxf-client, jetbrains-http-client, julia,
        k6, kotlin, lua, n4js, nim, objc, ocaml, perl, php, php-dt,
        php-nextgen, powerbuilder, powershell, python, r, ruby, rust,
        scala-akka, scala-gatling, scala-http4s-client, scala-pekko,
        scala-play-server, scala-sttp, scala-sttp4, scalaz, swift5,
        typescript-angular, typescript-aurelia, typescript-axios,
        typescript-fetch, typescript-inversify, typescript-jquery,
        typescript-nestjs, typescript-node, typescript-redux-query,
        typescript-rxjs, xojo-client, zapier
    ...
    ```

    To get a compact, comma-separated list suitable for scripting:

    ```bash
    openapi-generator-cli list -s | tr ',' '\n'
    ```

    To see configuration options for a specific generator, use `config-help`:

    ```bash
    openapi-generator-cli config-help -g typescript-axios
    ```
  </Step>

  <Step title="Generate a TypeScript Axios client">
    Generate a TypeScript client using the `typescript-axios` generator:

    ```bash
    openapi-generator-cli generate \
      -i petstore.yaml \
      -g typescript-axios \
      -o ./generated/typescript-client
    ```

    The `-i` flag specifies the input spec, `-g` is the generator name, and `-o` is the output directory.

    The generated directory will contain:

    ```
    generated/typescript-client/
    ├── api.ts           # Auto-generated API classes
    ├── base.ts          # Base configuration
    ├── common.ts        # Shared utilities
    ├── configuration.ts # Client configuration
    ├── index.ts         # Barrel export
    ├── package.json
    └── tsconfig.json
    ```

    You can pass additional generator-specific options using `--additional-properties`:

    ```bash
    openapi-generator-cli generate \
      -i petstore.yaml \
      -g typescript-axios \
      -o ./generated/typescript-client \
      --additional-properties=supportsES6=true,npmName=my-petstore-client,npmVersion=1.0.0
    ```

    <Tip>
      Run `openapi-generator-cli config-help -g typescript-axios` to see all available properties for the TypeScript Axios generator.
    </Tip>
  </Step>

  <Step title="Generate a Spring Boot server stub">
    Generate a Java Spring Boot server stub using the `spring` generator:

    ```bash
    openapi-generator-cli generate \
      -i petstore.yaml \
      -g spring \
      -o ./generated/spring-server
    ```

    The generated project is a runnable Spring Boot application with:

    ```
    generated/spring-server/
    ├── pom.xml
    └── src/
        └── main/
            ├── java/org/openapitools/
            │   ├── api/          # Generated API interfaces
            │   ├── model/        # Generated model classes
            │   └── OpenApiGeneratorApplication.java
            └── resources/
                └── application.properties
    ```

    The generated API interfaces contain method stubs with `// TODO: Implement` comments. Implement the methods in your own classes that extend the generated interfaces.

    To generate a server in a different framework, replace `spring` with another server generator:

    ```bash
    # Python FastAPI
    openapi-generator-cli generate -i petstore.yaml -g python-fastapi -o ./generated/fastapi-server

    # Go with Gin
    openapi-generator-cli generate -i petstore.yaml -g go-gin-server -o ./generated/go-server

    # Node.js
    openapi-generator-cli generate -i petstore.yaml -g nodejs-express-server -o ./generated/node-server
    ```
  </Step>

  <Step title="Validate your spec">
    Before generating code in a real project, validate your spec to catch errors early:

    ```bash
    openapi-generator-cli validate -i petstore.yaml
    ```

    A valid spec produces:

    ```text
    Validating spec (petstore.yaml)
    No validation issues detected.
    ```

    An invalid spec lists errors:

    ```text
    Validating spec (my-api.yaml)
    Errors:
        -attribute info is missing

    [error] Spec has 1 errors.
    ```

    Use `--recommend` to also receive improvement suggestions:

    ```bash
    openapi-generator-cli validate -i petstore.yaml --recommend
    ```
  </Step>
</Steps>

## Common options

Every `generate` command shares these core flags:

| Flag | Short | Description |
| ---- | ----- | ----------- |
| `--input-spec` | `-i` | Path or URL to the OpenAPI spec (required) |
| `--generator-name` | `-g` | Generator to use (required) |
| `--output` | `-o` | Output directory (required) |
| `--config` | `-c` | Path to a JSON/YAML config file |
| `--additional-properties` | `-p` | Generator-specific options as `key=value` pairs |
| `--skip-validate-spec` | | Skip spec validation before generation |
| `--skip-overwrite` | `-s` | Skip overwriting existing files |
| `--global-property` | | Set global properties (e.g., `models`, `apis`, `supportingFiles`) |

## Using a config file

For repeatable generation with many options, put your settings in a JSON config file:

```json
{
  "inputSpec": "petstore.yaml",
  "generatorName": "typescript-axios",
  "outputDir": "./generated/typescript-client",
  "additionalProperties": {
    "supportsES6": true,
    "npmName": "my-petstore-client",
    "npmVersion": "1.0.0"
  }
}
```

Then reference it with `-c`:

```bash
openapi-generator-cli generate -c openapi-config.json
```
